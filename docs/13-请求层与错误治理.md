# 请求层封装与前端错误治理

## 背景

在这次改动之前，前端请求层存在几个典型问题：

- 页面和表单里重复写 `fetch -> response.json() -> success/data/message` 解析逻辑。
- `401`、`404`、`500`、非 JSON 响应、超时没有统一语义。
- 浏览器运行时错误、请求错误、路由错误边界没有统一上报入口。
- 页面错误提示能展示，但排障信息和 traceId 不能稳定串起来。

这次改动的目标是把这些能力收敛成统一基础设施，而不是继续在业务页面里复制。

## 设计目标

这次请求层和错误治理要解决四件事：

```text
1. 统一解析 success / data / message / traceId
2. 统一处理非 JSON 响应和 JSON parse 失败
3. 统一处理超时、简单重试、401 登录跳转
4. 统一上报前端错误，包含 URL、用户、traceId
```

## 当前架构

这次实现把请求和错误治理拆成三层：

```text
业务组件层
  -> 请求封装层
  -> 页面边界层
  -> 错误上报层
```

## 四、常用 HTTP 错误类型

错误治理的第一原则是先判断责任边界：

```text
是谁的问题？
  |
  |-- 客户端请求有问题 -> 4xx
  |
  |-- 服务端处理出问题 -> 5xx
```

也就是：

```text
4xx = 你发来的请求，我不能或不应该处理
5xx = 你的请求可能没问题，但服务端没处理成功
```

### 总体判断图

```text
Browser / Client
      |
      | HTTP Request
      v
BFF / Server
      |
      |-- 请求不合法?          -> 400
      |-- 没登录?              -> 401
      |-- 登录了但没权限?       -> 403
      |-- 资源不存在?          -> 404
      |-- 请求太频繁?          -> 429
      |
      |-- 服务内部异常?         -> 500
      |-- 上游服务响应异常?     -> 502
      |-- 服务或依赖不可用?     -> 503
      |-- 上游超时?            -> 504
```

### 4xx：客户端侧错误

`4xx` 表示请求本身、身份、权限或目标资源有问题。客户端需要改请求、重新登录，或者换一个有权限的账号。

```text
Client Request
  |
  v
Server 校验请求
  |
  |-- body / query / param 不合法
  |       -> 400 Bad Request
  |
  |-- 没有登录态
  |       -> 401 Unauthorized
  |
  |-- 有登录态，但权限不够
  |       -> 403 Forbidden
  |
  |-- 要访问的数据不存在
  |       -> 404 Not Found
  |
  |-- 请求频率超过限制
          -> 429 Too Many Requests
```

| 状态码 | 含义         | 项目里的例子                                       |
| ------ | ------------ | -------------------------------------------------- |
| `400`  | 请求参数错误 | 商品创建缺字段、状态变更没填 reason、伪造 operator |
| `401`  | 未登录       | 没有 `next_bff_session` 访问商品接口               |
| `403`  | 无权限       | operator 访问审计日志                              |
| `404`  | 资源不存在   | 查询不存在的商品详情                               |
| `429`  | 请求太频繁   | 登录失败次数过多被限流                             |

### 为什么区分 401 和 403

```text
401: 我不知道你是谁
403: 我知道你是谁，但你不能做这件事
```

示例：

```text
没有 cookie
  -> 401

operator 已登录，但访问 admin-only 审计日志
  -> 403
```

这两个状态不能混用。否则前端无法判断应该跳登录页，还是展示无权限提示；排障时也无法判断问题是登录态丢失还是权限配置错误。

### 5xx：服务端侧错误

`5xx` 表示客户端请求看起来可以处理，但服务端、BFF 依赖或上游服务失败。

```text
Client Request 看起来没问题
      |
      v
Server 开始处理
      |
      |-- 代码异常 / 未捕获异常
      |       -> 500 Internal Server Error
      |
      |-- BFF 调 backend，backend 响应异常
      |       -> 502 Bad Gateway
      |
      |-- Mongo / Redis / backend 不可用
      |       -> 503 Service Unavailable
      |
      |-- BFF 等 backend 超时
              -> 504 Gateway Timeout
```

| 状态码 | 含义             | 适用场景                             |
| ------ | ---------------- | ------------------------------------ |
| `500`  | 服务端内部错误   | 未捕获异常、代码 bug、意外空值       |
| `502`  | 网关拿到异常响应 | BFF 调 backend，backend 返回异常格式 |
| `503`  | 服务暂不可用     | Mongo / Redis / backend 不可用       |
| `504`  | 上游超时         | BFF 等 backend 超时                  |

### 项目内统一错误结构

无论错误来自 Guard、Pipe、Controller 还是 Service，最终都会进入统一异常处理：

```text
Controller / Guard / Pipe / Service 抛异常
      |
      v
HttpExceptionFilter
      |
      v
统一错误响应
```

统一响应结构：

```json
{
  "success": false,
  "statusCode": 400,
  "message": "reason should not be empty",
  "path": "/api/commodity/10099/status",
  "traceId": "trace-status-missing-reason"
}
```

这样设计的原因：

- 前端不用猜不同接口的错误格式。
- e2e 测试可以稳定断言。
- `traceId` 可以串起前端、BFF、日志和审计。
- 用户提示和开发排障可以分开处理。

### 判断口诀

```text
请求本身错了吗?
  是 -> 400

用户没登录吗?
  是 -> 401

用户登录了但没权限吗?
  是 -> 403

资源不存在吗?
  是 -> 404

请求太频繁吗?
  是 -> 429

请求没问题，但服务炸了吗?
  是 -> 500

服务依赖的上游炸了或超时了吗?
  是 -> 502 / 503 / 504
```

## 六、非 JSON 响应与超时治理

### 非 JSON 响应

浏览器端和服务端 helper 都统一做了：

```text
先看 content-type
不是 application/json -> 不直接 JSON.parse
返回 envelope = null
走 fallbackMessage / AppError / ClientApiError
```

所以即使后端返回 HTML、空 body、网关错误页，也不会因为 `response.json()` 抛异常导致白屏。

### 超时

当前统一使用：

- 普通请求默认 `8s`
- 上传请求 `15s`

机制：

```text
AbortController
  -> 超时 abort
  -> 分类为 timeout error
  -> 返回用户可读提示
  -> 同时上报前端错误
```

图示：

```text
发起请求
  │
  ├─ 8s / 15s 内成功
  │    └─ 正常返回 data
  │
  └─ 超时
       ├─ AbortController abort
       ├─ 构造 timeout error
       ├─ 上报 frontend error
       └─ 页面显示“请求超时，请稍后重试”
```
