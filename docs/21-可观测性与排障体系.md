# 8. 可观测性与排障体系

真实系统上线后，必须能定位问题在哪一层，而不是只看到“页面报错”。当前系统用结构化日志、Metrics、Tracing 和前端上报组合成一套排障链路。

## 一、四类信号各自回答什么

结构化日志

回答：发生了什么？

`event / traceId / status / stack`

Metrics 指标

回答：整体健康吗？

`requestCount / errorRate / p95 / p99`

Tracing 链路

回答：这一次请求慢在哪里？

`BFF -> backend -> MongoDB`

前端上报

回答：用户在哪个页面、哪个版本遇到问题？

`page / appVersion / user / stack`

## 三、traceId 与 otelTraceId 如何串起来

当前系统保留两套 ID：`traceId`用于业务日志和前端排查，`otelTraceId`用于 tracing 平台调用树。

### traceId 关联图

```text

traceId=trace_xxx
BFF 结构化日志
otelTraceId=otel_trace_xxx

BFF -> backend -> MongoDB
排障顺序
前端 traceId -> 查 BFF 日志 -> 拿 otelTraceId -> 打开 tracing 调用树
```

## 四、三种典型排障路径

### 页面报 500

页面 error.tsx 显示 traceId

查 BFF 日志的 status / stack

用 otelTraceId 打开调用树

判断 BFF、backend、MongoDB 哪层报错

### 商品列表整体变慢

Metrics 出现 p95Ms 升高

查 alert_core_api_latency_high

看 tracing span duration

定位慢在 BFF、backend 或数据库

### 某个前端版本白屏

查 frontend_error_reported

看 page / appVersion / stack

用 source map 还原源码

如果有 traceId，再关联后端链路

## 五、慢请求调用树怎么读

调用树可以理解成“这一次请求的时间账本”。父 span 是外层动作，子 span 是里面发生的动作。父 span 的耗时通常包含子 span，所以排障时不是把所有耗时相加，而是看最长的子树在哪里。

读调用树的核心规则：先看最外层总耗时，再向下找占比最高的子 span。哪个子 span 接近总耗时，瓶颈通常就在那个分支。

### 慢请求调用树

```text
情况一：慢在数据库
BFF HTTP server span = 1200ms，总请求耗时
BFF -> backend = 1100ms，占总耗时 91%
Backend HTTP server span = 1050ms，慢点继续向下
MongoDB commodities list = 980ms，最大子 span
结论：查索引和查询
```

### 慢在 BFF 的调用树

```text
情况二：慢在 BFF 自身
BFF HTTP server span = 1200ms，总请求耗时
BFF -> backend = 80ms，只占 6%
结论：查 BFF 权限、审计、响应组装、等待或队列
```

| 看到的调用树 | 怎么判断 | 下一步查什么 |
| --- | --- | --- |
| `BFF -> backend` 接近总耗时 | BFF 大部分时间在等 backend | 继续看 backend 子 span、数据库 span、backend 错误日志 |
| `MongoDB commodities list` 接近 backend 耗时 | 慢点主要在数据库查询 | 查索引、查询条件、分页 offset、sort 字段 |
| `BFF -> backend` 很短，但 BFF 总耗时很长 | 慢点主要在 BFF 自己 | 查权限校验、审计写入、Redis、响应封装、同步阻塞逻辑 |

## 六、真实模拟案例

读案例前先记住：

`event`

是日志事件名，

`traceId`

是业务排查 ID，

`otelTraceId`

是 tracing 平台里的调用链 ID，

`span`

是调用链中的一个步骤，

`p95Ms`

是 95% 请求都不超过的耗时。

### Case 1：商品列表页面突然变慢

用户现象

运营反馈商品列表打开需要 2 秒以上，但页面最终能出来。

前端指标

`web_vital_metric`

显示

`page=/present/commodity/list`

的 LCP 变高。这里的 LCP 是最大内容绘制，用来衡量首屏主要内容多久出现。

BFF metrics

`http_request_metric`

显示

`route=GET /api/commodity/list`

，

`p95Ms=1800`

。p95Ms=1800 表示最近窗口里 95% 的商品列表请求耗时不超过 1800ms，已经偏慢。

告警事件

出现

`alert_core_api_latency_high`

， 这是“核心接口延迟过高”的告警事件，说明不是单个用户慢，而是商品列表接口整体变慢。

Tracing

调用树显示

`MongoDB commodities list=1500ms`

， 占 BFF 总耗时大部分。MongoDB commodities list 是数据库查询 span，代表商品查询和统计总数这一步。

结论：慢点在数据库查询。优先检查商品列表查询条件、排序字段、分页 offset 和 MongoDB 索引。

### Case 2：商品列表接口 5xx 错误率升高

用户现象

部分用户打开商品列表直接看到“页面加载失败”。

前端错误

`frontend_error_reported`

带有

`traceId=trace_500_001`

、

`page=/present/commodity/list`

。frontend_error_reported 是浏览器错误上报事件，traceId 是拿去查 BFF 日志的线索。

BFF 日志

查

`traceId`

，看到

`status=500`

、

`event=http_request_failed`

。500 表示服务端错误，不是参数错误或权限拒绝。

Metrics

窗口内

`serverErrorRate=0.2`

，触发

`alert_core_api_error_rate_high`

。serverErrorRate=0.2 表示 5xx 错误率是 20%。

Tracing

用日志里的

`otelTraceId`

打开 trace，看到错误发生在 backend 处理商品列表阶段。otelTraceId 是 OpenTelemetry 调用链 ID，用来在 Jaeger 或 Tempo 里查调用树。

结论：这是核心接口系统性 5xx，不是单个浏览器问题。先看 backend 异常日志和对应 span 的错误标记。

### Case 3：只有新版本前端审计页白屏

用户现象

管理员打开审计日志页白屏，商品列表页正常。

前端错误

`frontend_error_reported`

显示

`page=/present/commodity/audit`

、

`appVersion=2026.05.09`

。page 表示出错页面，appVersion 表示发布版本，用来判断是否是某次发布引入的问题。

堆栈

`stack`

指向审计页渲染逻辑；如果生产接入 source map，可以还原到具体源码行。stack 是运行时报错堆栈，source map 是把压缩后的生产代码映射回源码的文件。

后端关联

如果错误里有

`traceId`

，继续查 BFF；如果没有，说明可能是纯前端渲染错误。纯前端错误通常不一定有后端请求链路。

对比版本

旧版本没有同类错误，新版本集中出现，优先回看本次审计页改动。

结论：问题优先定位在前端版本变更。后端 trace 只用于排除接口错误，不是第一排查对象。

### Case 4：operator 访问审计页报 403

用户现象

operator 点击审计日志入口，页面提示无权限。

页面逻辑

审计页先执行

`getCurrentUser`

，发现

`roles`

不包含

`admin`

。roles 是当前登录用户的角色列表，admin 才允许查看审计日志。

AppError

页面抛出

`createAppError({ status: 403 })`

，进入

`error.tsx`

。AppError 是前端包装的业务错误，403 表示“知道你是谁，但你没有权限”。

前端上报

`frontend_error_reported`

记录

`page=/present/commodity/audit`

和用户信息。这里记录用户是为了确认是不是角色配置导致的预期拒绝。

判断

这是权限隔离的预期结果，不是系统故障。

结论：403 是业务权限控制，不应触发 5xx 告警；排障时要先区分“预期拒绝”和“系统异常”。

## 七、字段速查表

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `traceId` | BFF / backend / 前端错误上报 | 业务排障主线，前端拿到后可查后端日志 |
| `otelTraceId` | OpenTelemetry active span | 从日志跳到 tracing 平台调用树 |
| `event` | 所有结构化日志 | 按事件类型检索，例如 `http_request_metric` |
| `p95Ms / p99Ms` | HttpMetrics | 判断接口整体延迟是否劣化 |
| `page / appVersion` | 前端错误和 Web Vitals | 定位页面、发布版本和 source map |
