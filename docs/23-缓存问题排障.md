# 9.2 缓存问题排障

缓存排障的第一原则是先判断“请求有没有发生、到了哪一层、返回的数据来自哪里”。 本次补充了商品列表排障响应头、日志字段、预热脚本和排查文档，让旧数据问题可以按层定位。

### Network

看浏览器有没有发请求，以及响应头里有没有缓存状态。

### traceId

用`x-trace-id`串起 BFF 日志和 backend 调用。

### 降级

缓存异常时页面保留 loading、重试、刷新和兜底文案。

## 一、先解释概念

| 概念 | 简单解释 | 排障时怎么看 |
| --- | --- | --- |
| Network | 浏览器开发者工具里的请求列表。 | 没有请求行，说明问题可能在浏览器缓存、前端状态或页面没有刷新。 |
| response header | 服务端返回的响应头。 | 看 `x-trace-id` 、缓存状态、缓存来源。 |
| traceId | 一次请求的链路编号。 | Network 里有 traceId，日志里也有同一个 traceId，说明请求到达 BFF。 |
| 回源 | 缓存没命中或过期后，请求继续打到真实服务或数据库。 | 商品列表 header 显示 `source=backend` 说明本次回源。 |
| 预热 | 高峰前主动访问热点接口，把缓存先写好。 | 用脚本提前请求首页、热门筛选页，减少发布后第一波 miss。 |

## 二、当前新增的排障信号

商品列表响应现在会带这些 header。它们不改变业务数据，只用于判断旧数据到底来自哪一层。

| Header | 含义 | 如何判断 |
| --- | --- | --- |
| `x-trace-id` | 链路 ID。 | 拿它去查 BFF/backend 日志，判断请求有没有到服务端。 |
| `cache-control: no-store` | 告诉浏览器不要缓存这个商品列表 JSON。 | 如果仍然没请求，大概率是前端状态或页面没有重新触发取数。 |
| `x-cache-layer` | 当前由哪一层做业务缓存决策。 | 商品列表当前是 `bff-redis` 。 |
| `x-commodity-list-cache-state` | 本次列表缓存状态。 | `fresh` 、 `stale` 、 `miss` 。 |
| `x-commodity-list-cache-source` | 本次返回数据来源。 | `redis` 表示来自缓存； `backend` 表示已回源。 |
| `x-commodity-list-cache-refresh` | 是否触发后台刷新。 | `background` 表示 stale 已返回旧值并在后台刷新。 |
| `x-commodity-list-cache-key` | 缓存 key 的 hash，不暴露原始查询内容。 | 同一个 hash 说明命中的是同一类列表缓存。 |

## 三、旧数据来自哪一层

### 旧数据来源排查图

```text
Browser
有没有发请求
CDN
静态/图片
Next
RSC 是否重跑
BFF
traceId 日志
Redis
fresh/stale/miss
DB
最终事实
判断顺序：先看 Network 有没有请求，再看 x-trace-id 有没有到 BFF，最后看缓存 header 是 redis 还是 backend。
商品列表设置 no-store，正常情况下旧列表更可能来自 Redis stale、页面未刷新、或 DB/backend 事实未变。
```

## 四、“浏览器没发请求”和“服务端返回旧数据”的区别

| 现象 | 说明 | 排查动作 |
| --- | --- | --- |
| Network 没有新的商品列表请求 | 请求没离开浏览器或 Next 页面没有重新触发取数。 | 看是否执行了 `router.refresh()` ，或页面是否还在用旧状态。 |
| Network 有请求，但没有 `x-trace-id` | 请求可能没有到 BFF，或者被更前面的层处理。 | 检查代理、CDN、Next route handler、请求 URL 是否正确。 |
| 有 `x-trace-id` ，缓存来源是 `redis` | BFF 到了，但返回的是 Redis 中的列表。 | 看状态是 `fresh` 还是 `stale` ，再查写操作是否失效缓存。 |
| 有 `x-trace-id` ，缓存来源是 `backend` | BFF 已经回源，但 backend 或 DB 返回的事实仍然是旧的。 | 查 backend 日志、DB 数据、写操作是否真正成功。 |

## 五、“状态已变但列表没变”排查 checklist

### 状态已变列表没变排查 checklist

```text
1. 看页面
是否重新请求
2. 看 header
state/source
3. 查 traceId
BFF/backend 日志
4. 查 Redis
key 是否失效
5. 查 DB
事实是否已变
具体检查项
A. 写状态接口是否 200？审计日志是否记录了 status_change 和 reason？
B. 列表请求 header 是否 source=redis 且 state=stale/fresh？如果是，检查写后失效是否执行。
C. 如果 source=backend 仍旧，检查 backend 查询条件、软删除条件、DB 状态字段。
```

## 六、缓存穿透、击穿、雪崩

| 问题 | 本质 | 页面表现 | 应对方式 |
| --- | --- | --- | --- |
| 缓存穿透 | 大量不存在或非法条件一直绕过缓存打到 backend/DB。 | 搜索不存在商品很慢，404/空列表很多，backend 压力升高。 | 参数校验、空结果短缓存、限流、兜底文案。 |
| 缓存击穿 | 一个热点 key 过期瞬间，大量请求同时回源。 | 热门列表突然变慢，单个接口 P95/P99 抬高。 | stale 返回旧值、后台刷新、互斥锁、预热热点 key。 |
| 缓存雪崩 | 大量 key 同时过期，或 Redis 整体不可用。 | 多个页面同时变慢、5xx 增多、Redis 错误日志变多。 | TTL 加随机抖动、降级、限流、监控告警、预热。 |

## 七、前端降级体验

当前商品列表已经具备基础降级体验：

| 状态 | 页面能力 | 文件 |
| --- | --- | --- |
| 加载中 | 展示 skeleton 和“商品列表加载中”。 | `apps/client/app/present/commodity/list/loading.tsx` |
| 请求失败 | 展示错误、traceId、重新加载、清空筛选、返回登录页。 | `apps/client/app/present/commodity/list/error.tsx` |
| 缓存异常 | BFF 记录 Redis 错误，列表请求降级为直接查 backend。 | `apps/bff/src/commodity/commodity-cache.service.ts` |

## 八、发布或大促前缓存预热

新增`pnpm warmup:commodity-cache`，用于提前访问商品列表热点查询。 它会打印 status、traceId 和缓存响应头，方便确认预热是否成功。

```text
NEXT_BFF_SESSION="next_bff_session=你的会话值" \
BFF_PUBLIC_BASE_URL="http://127.0.0.1:3001" \
COMMODITY_CACHE_WARMUP_QUERIES="page=1&pageSize=20&sortBy=createdAt&sortOrder=desc|page=1&pageSize=20&status=on_sale" \
pnpm warmup:commodity-cache
```

预热不是为了绕过权限。脚本仍然需要登录态 cookie，因为商品列表接口本身受`AuthGuard`和`PermissionsGuard`保护。
