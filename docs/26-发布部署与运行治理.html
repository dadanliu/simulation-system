10. 发布、部署与运行治理

这一章解决的是“代码怎么从仓库变成线上服务，以及线上服务怎么证明自己还能接流量”。当前系统把 client、BFF、backend 分开构建；把构建时变量和运行时变量分开；用 version、commit sha、release note 追踪发布；用 live/ready 支持滚动发布和快速回滚。

构建边界

pnpm build:client、pnpm build:bff、pnpm build:server 分别产出三个可部署单元。

运行校验

APP_ENV=production 时，缺少发布变量或依赖不可用会失败，不会静默启动。

发布治理

/api/health/ready 控制是否接流量，错误率升高时用 commit sha 判断回滚目标。

一、当前构建过程怎么跑

根目录的 pnpm build:all 会让 workspace 中所有有 build 脚本的包各自构建。这里不是把三端打成一个大包，而是保留三个清晰产物边界。

命令 当前作用 真实例子
pnpm build:client 构建 Next.js 页面、RSC、API route、静态资源。 发布管理后台页面，页面顶部可展示版本号和 commit。
pnpm build:bff 构建 NestJS BFF，产物在 apps/bff/dist。 部署对外 API、登录会话、审计、上传代理、缓存和 health。
pnpm build:server 构建 NestJS backend mock service，产物在 apps/server/dist。 部署商品数据、文件存储、MongoDB 查询模拟层和 health。

二、构建时变量和运行时变量怎么区分

第一性原理：构建时变量会被写进构建产物或影响构建结果；运行时变量是在服务启动或处理请求时读取。当前系统的核心规则是： NEXT_PUBLIC_* 给浏览器可见，BFF/backend 的敏感配置只在服务端运行时读取。 

# client 生产必须显式提供
NEXT_PUBLIC_APP_ENV=production
NEXT_PUBLIC_APP_VERSION=2026.05.11.1
NEXT_PUBLIC_RELEASE_COMMIT_SHA=abc123def456
BFF_BASE_URL=https://bff.example.com
NEXT_INTERNAL_ORIGIN=https://admin.example.com

# BFF / backend 生产必须显式提供
APP_ENV=production
APP_VERSION=2026.05.11.1
RELEASE_COMMIT_SHA=abc123def456
MONGODB_URI=mongodb+srv://...
REDIS_URL=redis://...
RELEASE_NOTES_URL=https://github.com/example/next-bff/releases/tag/2026.05.11.1

三、版本追踪和回滚怎么工作

版本追踪的目标是：当线上错误率升高时，先知道“当前跑的是哪次发布”，再决定回滚到哪个 commit。当前系统在页面、BFF 根接口、backend 根接口和 health 返回中都带版本信息。 

// 页面顶部
v2026.05.11.1
abc123de
release

// BFF / backend 返回
{
  "service": "bff",
  "version": "2026.05.11.1",
  "commitSha": "abc123def456",
  "releaseNotesUrl": "https://github.com/example/next-bff/releases/tag/2026.05.11.1"
}

四、health check 和滚动发布怎么处理

live 和 ready 的区别很重要：live 说明进程还活着，ready 说明服务可以接业务流量。滚动发布时负载均衡器应该看 ready，而不是只看进程是否存活。

接口 当前检查什么 谁使用
BFF /api/health/live 只表示 BFF 进程存活。 进程级 liveness probe。
BFF /api/health/ready 检查 MongoDB、Redis、backend ready；失败返回 503。 负载均衡器、滚动发布、排障人员。
backend /api/health/live 只表示 backend 进程存活。 进程级 liveness probe。
backend /api/health/ready 检查 MongoDB；失败返回 503。 BFF ready 检查、负载均衡器、滚动发布。

五、真实 case：MongoDB 不可用时怎么表现

真实发布时最怕的是“服务进程启动了，但数据库连不上，流量已经打进来”。当前实现用启动前检查和 ready probe 两层拦住这个问题。 

// backend 启动阶段
app.init()
await app.get(HealthService).assertStartupReady()

// 如果 MongoDB 不可用
throw new Error("Backend startup dependency check failed: mongodb ...")

// 如果运行中 MongoDB 断开
GET /api/health/ready
HTTP 503
{
  "service": "backend",
  "status": "unready",
  "dependencies": [
    {
      "name": "mongodb",
      "status": "down",
      "message": "MongoDB connection is not ready, readyState=0"
    }
  ]
}

对用户来说，正确结果不是看到一堆 500，而是发布平台/负载均衡器发现 ready=503 后不再把新流量打给这个实例。
