# 15. 测试、CI 与质量门禁

本文记录 `7. 测试、CI 与质量门禁` 的落地方案。

目标不是追求测试数量，而是保证登录、鉴权、权限、响应封装、异常过滤、商品主链路这些核心路径不会被改坏。

## 15.1 图例

```text
[Test]          Jest / Supertest 自动化测试
[TestingModule] NestJS 测试模块
[Guard]         AuthGuard / PermissionsGuard
[Filter]        HttpExceptionFilter
[Interceptor]   SuccessResponseInterceptor
[CI]            GitHub Actions 质量门禁
```

## 15.2 测试分层

当前 BFF 测试分两类：

```text
Unit Test
  - Service 内部规则
  - 密码 hash
  - 登录风控
  - 审计日志查询
  - 文件 URL 规则

E2E Test
  - Controller + Guard + Pipe + Filter + Interceptor
  - Supertest 发起真实 HTTP 请求
  - Cookie / CSRF / traceId / 响应 envelope 断言
```

测试入口：

```text
pnpm test:bff
  -> apps/bff package test
  -> jest --config ./jest.config.js --runInBand

pnpm test:bff:e2e
  -> apps/bff package test:e2e
  -> 只运行 *.e2e-spec.ts
```

相关文件：

- `apps/bff/package.json`
- `apps/bff/jest.config.js`
- `apps/bff/src/test/app-test-utils.ts`

## 15.3 核心 e2e 覆盖

### 登录成功链路

覆盖文件：

- `apps/bff/src/auth/auth.e2e-spec.ts`

覆盖点：

```text
[Test]
  POST /api/auth/login
      |
      v
[AuthController]
  CSRF 校验
  AuthService.login
  Set-Cookie
      |
      v
[Interceptor]
  success envelope
```

断言重点：

- 返回 `201`。
- 响应体包含 `success / data / message / traceId`。
- 写入 `next_bff_session`。
- Cookie 包含 `HttpOnly / SameSite / Max-Age`。
- 生产环境下正确追加 `Secure`。

### 未登录访问商品接口

覆盖文件：

- `apps/bff/src/commodity/commodity.e2e-spec.ts`

链路：

```text
GET /api/commodity/list
  无 session
      |
      v
[AuthGuard]
  currentUser = null
      |
      v
[Filter]
  401 unified response
```

断言重点：

- 返回 `401`。
- 响应体包含 `statusCode / success=false / traceId / path`。
- 商品 service 不会被调用。

### 无权限访问

覆盖文件：

- `apps/bff/src/commodity/commodity.e2e-spec.ts`

覆盖场景：

```text
operator 创建商品但权限服务返回 false
operator 编辑商品但权限服务返回 false
operator 查看审计日志
非 admin 即使 audit:read 通过也不能查看审计日志
```

链路：

```text
[AuthGuard]
  已登录
      |
      v
[PermissionsGuard]
  hasAllPermissionsByRoleCodes = false
      |
      v
[Filter]
  403 unified response
```

断言重点：

- 返回 `403`。
- 响应 message 为 `permission denied`。
- 业务 service 不会被调用。

### 商品创建、状态变更、删除

覆盖文件：

- `apps/bff/src/commodity/commodity.e2e-spec.ts`

成功路径：

```text
POST /api/commodity/create
PATCH /api/commodity/:id/status
DELETE /api/commodity/:id
```

失败路径：

```text
400  DTO 参数非法
400  状态变更缺 reason
400  删除商品缺 reason
400  前端伪造 operator
401  未登录访问
403  无权限访问
404  商品详情不存在
```

## 15.4 为什么 e2e 可以防 Guard / Filter / Interceptor 回归

这些测试不是直接调用 controller 方法，而是用 Supertest 发起 HTTP 请求：

```text
Supertest
  -> Nest HTTP Pipeline
  -> Middleware
  -> Guard
  -> Pipe
  -> Controller
  -> Service mock
  -> Interceptor
  -> Filter
```

所以如果改坏下面任意一层，主链路测试会失败：

- `AuthGuard` 没有正确注入 `currentUser`。
- `PermissionsGuard` 没有正确读取权限元数据。
- `ValidationPipe` 没有返回 400。
- `HttpExceptionFilter` 没有统一错误结构。
- `SuccessResponseInterceptor` 没有统一成功结构。
- CSRF 中间件错误拦截写操作。

## 15.5 CI 门禁

核心质量门禁 workflow：

- `.github/workflows/bff-quality-gate.yml`
- `.github/workflows/client-e2e.yml`

触发时机：

```text
pull_request
push main
push master
```

`bff-quality-gate.yml` 虽然保留历史文件名，但现在是全仓库核心门禁，拆成三个 job：

```text
[CI Core Quality Gate]
  checkout
  setup pnpm 10.23.0
  setup Node.js 22
  pnpm install --frozen-lockfile

  lint job
    -> pnpm lint:all

  bff-test job
    -> pnpm test:bff

  build job
    -> pnpm build:all
```

质量门禁含义：

| Job        | 命令             | 目的                                      |
| ---------- | ---------------- | ----------------------------------------- |
| `lint`     | `pnpm lint:all`  | 全仓库 ESLint，覆盖 client / BFF / server |
| `bff-test` | `pnpm test:bff`  | Jest 单测 + BFF e2e，覆盖后端主链路       |
| `build`    | `pnpm build:all` | Next build + Nest build + TypeScript 检查 |

`client-e2e.yml` 负责浏览器真实链路：

```text
[CI Client E2E]
  MongoDB service
  Redis service
  pnpm exec playwright install --with-deps chromium
  pnpm test:client:e2e
  upload playwright-report / test-results/playwright
```

CI 页面会按 job 名直接显示失败类型：

```text
Lint all packages
Test BFF
Build all packages
Playwright core user journeys
```

这样 lint、build、test 任一失败都会让对应 job 失败。仓库开启分支保护后，把这些 workflow 作为 required checks，就能阻断 PR 合并。

## 15.6 暂不加入 format:check 的原因

当前执行：

```bash
pnpm format:check
```

会因为历史文件格式不一致失败，范围超过本次 7.1 的测试/CI 主链路目标。

因此本次 CI 不加入全仓库格式检查，避免新门禁一上线就因为历史格式问题变红。

后续如果要启用格式门禁，建议单独做一次：

```text
1. 全仓库 prettier --write
2. 单独提交格式化变更
3. 再把 pnpm format:check 加入 CI
```

## 15.8 前端 e2e 测试

新增 Playwright 覆盖后台核心前端链路：

```text
[Playwright]
  -> 启动 e2e services
  -> 重置测试数据
  -> 浏览器操作页面
  -> 断言 UI 与审计结果
```

一键入口：

```bash
pnpm test:client:e2e
```

本地调试入口：

```bash
pnpm test:client:e2e:headed
```

相关文件：

- `playwright.config.ts`
- `scripts/e2e-services.js`
- `e2e/commodity.spec.ts`
- `.github/workflows/client-e2e.yml`

### 测试环境图例

```text
Playwright
  |
  v
Client 127.0.0.1:3200
  |
  v
BFF 127.0.0.1:3201
  |
  v
Mock Server 127.0.0.1:3202
  |
```

端口与数据库使用独立测试配置，避免污染开发环境：

| 服务        | 测试地址                        | 说明             |
| ----------- | ------------------------------- | ---------------- |
| Client      | `http://127.0.0.1:3200`         | Next.js 页面     |
| BFF         | `http://127.0.0.1:3201`         | Cookie / API     |
| Mock Server | `http://127.0.0.1:3202`         | 商品与上传后端   |
| MongoDB     | `127.0.0.1:27018/next-bff-test` | 可重复初始化数据 |
| Redis       | `redis://127.0.0.1:6379`        | 会话与风控缓存   |

### 覆盖链路

```text
登录后访问商品列表
  admin 登录
  -> /present/commodity/list
  -> 断言默认商品可见

创建带图商品后列表展示图片
  admin 登录
  -> 上传 PNG
  -> 创建商品
  -> 回到列表
  -> 断言商品行存在 img

operator 修改状态，admin 查看审计
  operator 登录
  -> 修改 10002 状态并填写 reason
  -> admin 登录
  -> 审计页按 action + targetId 查询
  -> 断言 operator / 状态变更 / reason / traceId

admin 删除商品后列表不可见
  admin 登录
  -> 商品详情填写 reason 和商品名确认
  -> 接受二次确认弹窗
  -> 回到列表
  -> 断言商品链接不存在
```

### 测试数据可重复初始化

BFF 新增仅测试环境开放的接口：

```text
POST /api/test/reset
```

开启条件：

```text
APP_ENV=test
或 E2E_TEST_RESET_ENABLED=true
```

重置内容：

- BFF RBAC 种子数据。
- 商品审计日志。
- 登录审计日志。
- Mock Server 商品种子数据。
- Mock Server 上传文件 registry。

这个接口不依赖前端传 operator，也不会在生产环境暴露。测试每个用例开始前调用一次，所以创建、状态变更、删除互不污染。

## 15.9 7.3 CI 质量门禁验收

图例：

```text
PR / push
  |
  +-- Core Quality Gate
  |     +-- Lint all packages
  |     +-- Test BFF
  |     +-- Build all packages
  |
  +-- Client E2E
        +-- Playwright core user journeys
        +-- Upload failure screenshots / trace / report
```

验收对应关系：

| 验收项             | 落地方式                                       |
| ------------------ | ---------------------------------------------- |
| lint 失败无法合并  | `Lint all packages` job 执行 `pnpm lint:all`   |
| build 失败无法合并 | `Build all packages` job 执行 `pnpm build:all` |
| test 失败无法合并  | `Test BFF` 和 `Playwright core user journeys`  |
| 快速定位失败项目   | workflow 拆成独立 job，Playwright 上传报告     |

本地等价验证命令：

```bash
pnpm lint:all
pnpm test:bff
pnpm build:all
pnpm test:client:e2e
```
