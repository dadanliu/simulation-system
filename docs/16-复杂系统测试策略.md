# 16. 复杂系统测试策略

真实复杂系统不能靠一种测试保证稳定，而是靠分层测试、CI 门禁和线上可观测性一起兜底。

核心原则：

```text
越底层
  数量越多
  速度越快
  定位越准

越上层
  数量越少
  越接近真实用户
  维护成本越高
```

## 16.1 总体图例

```text
                手工验收 / 探索测试
              ─────────────────────
                 浏览器 E2E 测试
              Playwright / Cypress
            ─────────────────────────
                 API / 集成测试
            Supertest / Testcontainers
          ─────────────────────────────
                   单元测试
              Jest / Vitest / Service Test
        ─────────────────────────────────
                   静态检查
          TypeScript / ESLint / Build / Format
```

这不是说上层测试不重要，而是说上层测试应该聚焦核心链路。复杂系统里最健康的测试结构通常是：

```text
大量静态检查和单元测试
中量 API / 集成测试
少量关键 E2E 测试
必要的手工探索和线上监控
```

## 16.2 静态检查

静态检查不启动完整系统，主要挡低级错误。

```text
代码提交
  |
  v
TypeScript / ESLint / Build
  |
  +-- 类型错误
  +-- 未使用变量
  +-- import 错误
  +-- 构建失败
  +-- 明显规范问题
```

当前项目对应命令：

```bash
pnpm lint:all
pnpm build:all
```

适合发现：

- TypeScript 类型不匹配。
- React / Nest 编译失败。
- ESLint 规则失败。
- import 路径错误。
- CI 里无法构建的页面或服务。

它不适合发现：

- 业务规则是否正确。
- 用户是否能完成登录。
- 权限是否真的挡住了接口。

## 16.3 单元测试

单元测试关注一个函数、一个 service 或一段业务规则。

```text
输入
  |
  v
业务函数 / Service 方法
  |
  v
输出 / 异常 / 副作用断言
```

适合测试：

- 权限判断。
- 状态机流转。
- DTO 转换。
- 审计 diff。
- 密码 hash。
- 登录风控。
- 验证码正确、错误、过期。
- 文件类型校验。

当前项目例子：

```text
apps/bff/src/auth/login-risk.service.spec.ts
apps/bff/src/auth/rbac-seed.service.spec.ts
apps/bff/src/commodity/audit-log.service.spec.ts
apps/bff/src/commodity/commodity.service.spec.ts
apps/bff/src/upload/file-url.service.spec.ts
```

图例：

```text
CommodityService
  |
  +-- 创建商品缺 name -> 400
  +-- 状态 pending -> on_sale -> 允许
  +-- 状态 offline -> pending -> 拒绝
  +-- 删除缺 reason -> 400
```

优点：

- 快。
- 稳定。
- 失败定位清楚。
- 适合覆盖大量边界条件。

缺点：

- 不验证真实 HTTP pipeline。
- 不验证浏览器页面。
- mock 过多时可能和真实系统脱节。

## 16.4 API / BFF E2E 测试

API e2e 不打开浏览器，而是直接发 HTTP 请求。

当前项目使用：

```text
Jest + Supertest
```

图例：

```text
Supertest
  |
  v
Nest HTTP Pipeline
  |
  +-- Middleware
  +-- Guard
  +-- Pipe
  +-- Controller
  +-- Service
  +-- Interceptor
  +-- Filter
```

适合测试：

- 登录接口成功。
- Cookie 是否写入。
- 未登录返回 401。
- 无权限返回 403。
- DTO 非法返回 400。
- 商品创建、编辑、状态变更、删除。
- 审计日志是否写入。
- traceId 是否返回。
- Filter / Interceptor 是否保持统一响应结构。

当前项目对应文件：

```text
apps/bff/src/auth/auth.e2e-spec.ts
apps/bff/src/commodity/commodity.e2e-spec.ts
```

它能防止这类回归：

```text
AuthGuard 改坏
  -> 未登录接口没有返回 401

PermissionsGuard 改坏
  -> operator 越权创建商品

ValidationPipe 改坏
  -> 非法参数没有返回 400

HttpExceptionFilter 改坏
  -> 错误响应结构不统一

SuccessResponseInterceptor 改坏
  -> 成功响应没有 success / data / traceId
```

## 16.5 集成测试

集成测试关注多个真实组件之间能不能协作。

图例：

```text
Service A
  |
  v
Redis / MongoDB / Queue / Storage
  |
  v
Service B
```

适合测试：

- MongoDB 查询条件是否正确。
- Redis key 过期是否正确。
- 上传文件是否能写入 storage。
- 审计日志是否真实入库。
- 事务失败是否回滚。
- 消息队列 consumer 是否正确处理任务。

复杂系统里很多问题不是函数错，而是组件协作错：

```text
代码逻辑没错
  |
  v
Redis TTL 配错
  |
  v
登录风控一直不解锁
```

或者：

```text
上传 API 返回成功
  |
  v
文件 registry 没写入
  |
  v
列表图片无法展示
```

## 16.6 前端组件测试

前端组件测试关注单个组件或局部交互，不一定启动完整后端。

常用工具：

```text
React Testing Library
Vitest / Jest
Storybook interaction test
```

适合测试：

- 表单校验。
- 按钮 disabled 状态。
- 错误提示展示。
- 空状态展示。
- 权限隐藏按钮。
- 弹窗打开关闭。
- 列表筛选 UI。

图例：

```text
CommodityDeleteForm
  |
  +-- reason 为空 -> 显示错误
  +-- 商品名确认不一致 -> 显示错误
  +-- 点击删除 -> 调用 deleteCommodity
```

它比 Playwright 更快，但不覆盖完整后端链路。

## 16.7 浏览器 E2E 测试

浏览器 e2e 使用真实浏览器模拟用户。

当前项目使用：

```text
Playwright
```

图例：

```text
Playwright
  |
  v
Chromium 浏览器
  |
  v
Next Client
  |
  v
Nest BFF
  |
  v
Mock Server
  |
  v
MongoDB / Redis
```

当前项目覆盖链路：

```text
登录后访问商品列表
创建带图商品后列表展示图片
operator 修改状态，admin 查看审计
admin 删除商品后列表不可见
```

适合测试：

- 真实页面跳转。
- 登录 cookie 会话。
- 表单填写。
- 文件上传。
- 图片展示。
- 二次确认弹窗。
- 多角色切换。
- 权限页面体验。
- 审计日志跨账号验证。

不适合测试：

- 每一个 DTO 字段。
- 每一种权限组合。
- 每一个 service 分支。
- 算法边界条件。
- 大量纯函数逻辑。

原因：

```text
Playwright 更接近真实用户
  |
  +-- 优点：真实
  +-- 代价：慢、依赖服务启动、失败原因更多
```

## 16.8 验证码场景怎么测

验证码的目标就是阻止自动化，所以 e2e 不应该尝试破解真实验证码。

推荐做法：

```text
生产环境
  -> 使用真实验证码 provider

测试环境
  -> 使用 FakeCaptchaProvider
  -> 固定验证码 1234
  -> 或关闭验证码，只测试登录主链路
```

图例：

```text
Playwright
  |
  v
填写 username / password / 1234
  |
  v
BFF 校验验证码
  |
  v
APP_ENV=test 时 FakeCaptchaProvider 返回通过
```

验证码本身应该用 API / 单元测试覆盖：

```text
验证码为空 -> 400
验证码错误 -> 400 / 401
验证码过期 -> 400
验证码正确 -> 通过
```

不推荐：

- OCR 识别图片验证码。
- 模拟滑块轨迹破解验证码。
- 调第三方打码平台。
- 在 CI 中依赖真实短信验证码。

## 16.9 回归测试

回归测试来自真实 bug。

流程：

```text
线上或测试环境发现 bug
  |
  v
写一个能复现 bug 的测试
  |
  v
修代码
  |
  v
测试长期留在 CI
```

价值：

```text
同一个 bug 不应该出现第二次
```

例如：

```text
问题：operator 可以伪造 body.operator 写审计
  |
  v
补测试：前端传 operator 字段仍然以登录态用户为准
  |
  v
以后改审计逻辑时自动防回归
```

## 16.10 契约测试

契约测试关注接口结构是否保持一致。

图例：

```text
Client 期待
  {
    id,
    name,
    imageUrl
  }

Server 返回
  {
    id,
    title,
    image_url
  }

契约失败
```

适合：

- 前后端分离。
- 微服务调用。
- 第三方 API。
- OpenAPI schema。
- JSON Schema。

常见工具：

```text
Pact
OpenAPI validation
JSON Schema
```

## 16.11 性能和安全测试

功能正确不代表系统稳定。

性能测试关注：

- QPS。
- P95 / P99 响应时间。
- 慢查询。
- 大分页。
- 并发登录。
- 大文件上传。
- Redis 热 key。

安全测试关注：

- 未登录访问。
- 越权访问。
- CSRF。
- XSS。
- NoSQL 注入。
- 文件上传校验。
- 敏感字段泄露。
- 审计日志不可伪造。

图例：

```text
安全测试
  |
  +-- 未登录 -> 401
  +-- 无权限 -> 403
  +-- 非法参数 -> 400
  +-- 伪造 operator -> 服务端忽略
  +-- 上传伪装图片 -> 拒绝
```

## 16.12 CI 门禁怎么组合

推荐 PR 级别门禁：

```text
PR
  |
  +-- pnpm lint:all
  +-- pnpm test:bff
  +-- pnpm build:all
  +-- pnpm test:client:e2e
```

当前项目 workflow：

```text
.github/workflows/bff-quality-gate.yml
  |
  +-- Lint all packages
  +-- Test BFF
  +-- Build all packages

.github/workflows/client-e2e.yml
  |
  +-- Playwright core user journeys
  +-- Upload Playwright report
```

GitHub 分支保护里把这些 checks 设置成 required 后：

```text
任一 job 失败
  |
  v
PR 不能合并
```

## 16.13 当前项目推荐测试分布

| 层级         | 工具                       | 当前项目应该覆盖                         |
| ------------ | -------------------------- | ---------------------------------------- |
| 静态检查     | TypeScript / ESLint        | 全仓库 lint、client / BFF / server build |
| 单元测试     | Jest                       | 权限、状态机、审计 diff、登录风控        |
| API e2e      | Jest + Supertest           | 登录、401、403、400、商品写操作          |
| 集成测试     | 测试 DB / Redis / 存储     | seed、审计入库、上传文件 registry        |
| 浏览器 e2e   | Playwright                 | 登录、带图创建、状态变更、审计、删除     |
| 回归测试     | 对应层级补测               | 每个真实 bug 留一个测试                  |
| 性能安全测试 | k6 / 安全扫描 / 自定义脚本 | 慢查询、越权、CSRF、上传安全             |

## 16.14 最终判断标准

测试不是越多越好，而是要回答四个问题：

```text
1. 这个功能坏了，哪个测试会第一时间发现？
2. 测试失败后，能不能快速定位是哪一层坏了？
3. 这个测试是否稳定，能不能在 CI 长期运行？
4. 这个测试维护成本是否小于它防住的问题成本？
```

一句话总结：

```text
静态检查挡低级错误
单元测试守业务规则
API 测试守接口契约
集成测试守组件协作
E2E 测试守核心用户路径
CI 门禁阻止坏代码合并
线上监控发现测试没有覆盖的问题
```
