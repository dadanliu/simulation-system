9.1 多层缓存设计

本次实现的目标不是“所有东西都缓存”，而是把缓存放在正确的层级： 静态资源和图片尽量靠近用户缓存；商品列表允许短缓存；权限、审计、当前登录态保持强一致； 商品写操作后主动失效列表缓存，避免创建、删除、恢复后长期看到旧数据。

商品列表

BFF 增加 Redis 短缓存，默认 fresh TTL 为 5s，stale 窗口为 30s。

图片和静态资源

图片走已签名 URL、缓存版本、ETag 和 CDN 缓存头；Next 静态资源依赖构建 hash 实现长期缓存。

安全数据

审计日志、权限、当前用户信息不做公共长缓存，避免权限变更后仍看到越权入口。

一、先解释概念

缓存文档里最容易让人误解的是：同一个“缓存”词，可能发生在浏览器、CDN、Next、BFF 或 Redis。位置不同，解决的问题也不同。

概念 简单解释 在当前系统里的含义
Browser Cache 浏览器自己的 HTTP 缓存。命中时，请求不一定会真正打到服务端。 适合缓存 JS、CSS、图片；不适合缓存当前用户权限、审计数据。
CDN 离用户更近的边缘节点，用来缓存静态资源或图片，减少源站压力。 适合放图片、Next 静态构建资源；不负责判断用户权限。
Cache-Control HTTP 响应头，告诉浏览器或 CDN 这个响应能不能缓存、缓存多久。 图片接口会返回缓存头；商品 JSON 页面取数保持谨慎。
ETag / Last-Modified 协商缓存标识。浏览器带着旧标识询问服务端：我本地这份还能不能用。 图片代理可用它们返回 304，避免重复传输图片 body。
资源 hash 把文件内容摘要放进文件名。内容变了，文件名也变。 Next 构建后的 JS/CSS 适合长期缓存，因为新版本会有新 URL。
签名 URL URL 里带服务端签出来的访问凭证，防止任意构造私有资源地址。 图片访问可以既可控又能被 CDN 缓存。
缓存版本化 通过版本参数或新 URL 让旧缓存自然失效。 商品换图后，图片 URL 变化，浏览器和 CDN 不会一直用旧图。
Redis 服务端内存型存储，常用于热点数据缓存。 BFF 用它短缓存商品列表；Redis 不是事实来源，DB 才是事实来源。
TTL Time To Live，缓存存活时间。到期后缓存应该被淘汰或视为过期。 商品列表默认 fresh TTL 是 5s。
fresh / stale / miss fresh 是新鲜缓存；stale 是过期但还能短暂使用的旧缓存；miss 是没有命中缓存。 BFF fresh 直接返回；stale 先返回旧值并后台刷新；miss 查 backend。
stale-while-revalidate 先用旧缓存保证响应速度，同时后台重新验证或刷新缓存。 商品列表 stale 窗口默认 30s，用于降低短时间抖动。
cache key 缓存里的唯一键。key 设计错了，就可能读到别人或别的条件的数据。 当前用 角色集合 + backendPath 组成商品列表 key。
缓存失效 主动删除或废弃旧缓存，让下一次读取重新查真实数据。 商品创建、编辑、状态变更、删除、恢复后都会失效列表缓存。
router.refresh() Next App Router 的刷新动作，让当前路由重新执行服务端取数。 前端写操作成功后刷新页面；真正的服务端缓存失效由 BFF 完成。
no-store 告诉 fetch 或 HTTP 缓存不要复用旧响应，每次都重新请求。 当前用户、权限、审计等可信数据保持 no-store 思路。
强一致 写入后，后续读取必须尽快看到最新事实，不能长期读旧值。 权限和审计偏强一致；商品列表可接受很短缓存，但写后必须主动失效。
软删除 不是物理删除数据库记录，而是标记 deletedAt 等字段。 商品删除后列表应过滤掉软删除商品，同时删除操作会写审计。
scan + del Redis 里按模式扫描 key，再删除命中的 key。 BFF 用它删除所有商品列表缓存 key。
DB Database，数据库。它保存业务最终事实。 MongoDB 保存商品、用户、权限、审计；缓存只是加速读取。

变更点 逻辑 为什么这样做
commodity-cache.service.ts 集中处理 Redis 连接、缓存读写、缓存失效、异常降级。 避免把 Redis 细节散落在业务服务里，后续替换缓存实现更容易。
listCommodities 先生成 backend 查询路径，再用用户角色和路径生成 cache key。 同一角色、同一筛选条件可以复用缓存；不同条件不会串数据。
fresh 命中 直接返回 Redis 中的列表。 减少 backend 和数据库压力，列表页打开更快。
stale 命中 先返回旧列表，同时后台刷新 Redis。 让用户先看到可用页面，后台再补新缓存。
miss 请求 backend 拿真实列表，再写回 Redis。 缓存没有命中时，数据库链路仍然是最终来源。
写操作后失效 商品创建、编辑、状态变更、删除、恢复成功后删除列表缓存。 避免列表在 TTL 内继续展示旧商品状态。
Redis 异常降级 Redis 读写失败时写结构化日志，请求继续走 backend。 缓存只提升性能，不应该影响主业务可用性。

五、Cache-Control、ETag、CDN、资源 hash 对比

这四个方案解决的是不同问题：Cache-Control 是缓存规则，ETag 是校验机制，CDN 是缓存位置，资源 hash 是版本化方式。真实项目里通常组合使用，而不是四选一。

方案 它控制什么 优点 限制 适合当前系统的哪里
Cache-Control 控制浏览器或 CDN 能不能缓存、缓存多久、过期后能不能用旧值。 标准 HTTP 能力，浏览器和 CDN 都支持，配置成本低。 如果给动态或权限数据配置过长，会导致用户长期看到旧数据。 图片文件响应、Next 静态资源响应。
ETag 控制“旧资源还能不能复用”。浏览器带旧 ETag 问服务端，没变就返回 304。 资源没变化时不传输 body，省带宽，适合大文件。 仍然会有一次请求到服务端或 CDN，不是完全不发请求。 商品图片、文件代理、可协商缓存的大资源。
CDN 控制资源缓存在哪。CDN 把资源放到离用户更近的边缘节点。 访问更快，源站压力更小，适合高频读取的大资源。 不能随便缓存用户私有数据、权限数据和审计数据。 图片、JS、CSS、公开静态文件。
资源 hash 控制资源 URL 是否随内容变化。内容变，文件名或 URL 也变。 可以放心长缓存，因为新版本会用新 URL。 适合构建产物，不适合每次都动态生成的业务 JSON。 Next 构建后的 JS/CSS，图片换图后的版本化 URL。

对商品列表来说，这四种都不是最合适的主方案。商品列表是动态业务 JSON， 会被创建、删除、状态变更影响，所以当前采用 BFF + Redis 短缓存，并在写操作后主动失效。 对权限和审计来说，优先级是可信和及时，因此不能用 CDN 长缓存。

七、商品列表短缓存如何工作

缓存 key 不是简单的 commodity:list，而是由 用户角色集合 + backendPath 组成。这样同一个筛选条件可以复用缓存， 同时避免不同角色看到不该共享的数据。

  • COMMODITY_LIST_CACHE_TTL_SECONDS=5：缓存处于 fresh 状态的时间。
  • COMMODITY_LIST_CACHE_STALE_SECONDS=30：fresh 过期后允许短暂返回旧值并后台刷新。
  • REDIS_URL：Redis 不可用时，BFF 会记录错误日志并自动降级为直查 backend。

八、cacheKey 如何新增、过期、删除

当前实现里，cacheKey 不是前端传来的，也不是固定写死的字符串。 它由 BFF 服务端根据“用户角色集合”和“转发给 backend 的查询路径”生成。 这样同一个角色、同一个筛选条件可以复用缓存；不同角色或不同筛选条件会得到不同 key。

1. cacheKey 怎么新增

列表请求进入 listCommodities 后，BFF 先把前端查询条件转换成 backendPath。然后 CommodityCacheService.readCommodityList(user, backendPath) 会调用 buildCommodityListKey 生成 key。

组成部分 示例 目的
key prefix next-bff:commodity:list: 标记这是商品列表缓存，便于后续按前缀批量删除。
排序后的角色集合 admin、operator 避免不同角色复用同一份列表缓存。
backendPath /api/commodity/list?limit=20&offset=0 不同分页、筛选、排序条件得到不同缓存。

真正“新增 key”发生在 miss 后请求 backend 成功，或者 stale 后后台刷新成功时。 BFF 会调用 writeCommodityList(cacheKey, data)，写入这样的结构：

  • data：商品列表响应数据。
  • freshUntil：当前时间加 fresh TTL，默认是当前时间 + 5s。
  • Redis 过期时间：freshTtlSeconds + staleTtlSeconds，默认 5 + 30 = 35s。

2. cacheKey 怎么过期

当前有两层过期逻辑，一层是业务状态过期，一层是 Redis 物理过期。

阶段 判断方式 行为
fresh freshUntil >= Date.now() 缓存仍新鲜，BFF 直接返回 Redis 里的列表。
stale freshUntil < Date.now()，但 Redis key 还在 先返回旧列表，同时后台请求 backend 刷新这条 key。
miss Redis 已经没有这条 key 请求 backend，拿到真实列表后重新写入 Redis。

所以 fresh TTL 到期不等于 Redis key 立刻消失。默认 5s 后只是从 fresh 变成 stale；默认 35s 后 Redis 的 EX 到期，key 才会被 Redis 自然删除。

3. cacheKey 怎么删除

当前有三种删除路径：

删除路径 代码行为 触发场景
Redis 自然删除 redis.set(key, value, "EX", freshTtl + staleTtl) 默认 35 秒后，如果没有刷新，Redis 自动删除该 key。
写操作主动删除 scan MATCH next-bff:commodity:list:* COUNT 100 后对命中的 key 执行 del。 创建、编辑、状态变更、删除、恢复商品成功后。
坏缓存单 key 删除 读取到无法解析或结构不对的缓存时，执行 del(key)。 Redis 里有脏数据、旧版本数据或手工写入了错误内容。

写操作用前缀批量删除，而不是只删当前分页的 key。原因是一次商品变化可能影响多个列表： 第一页、第二页、不同筛选条件、不同排序、不同状态 tab 都可能受影响。当前系统选择简单可靠的方式： 只要商品发生写变化，就删除所有商品列表缓存。

九、写操作后的缓存失效

前端的 router.refresh() 只负责让当前页面重新取数；真正保证旧列表不会长期存在的是 BFF 写成功后的 invalidateCommodityList()。两个动作配合后，用户回到列表时会重新走 BFF 读流程，并在 Redis miss 后拿到 backend 的新数据。

十、真实场景

场景 A：创建商品后列表没有新商品

如果只做 TTL，不主动失效，用户创建商品后可能在几秒到几十秒内继续看到旧列表。 当前系统在 createCommodity 写成功并记录审计后，会删除商品列表 Redis key。 下一次访问列表时 Redis miss，BFF 重新请求 backend，所以新商品会出现。

场景 B：admin 删除商品后列表仍可见

删除是软删除，backend 会改变商品状态或删除字段。BFF 在 deleteCommodity 成功后失效列表缓存；前端删除表单跳转到 /present/commodity/list 并刷新页面，因此列表不会长期展示已删除商品。

场景 C：operator 权限被收回后菜单还在

权限和当前用户信息没有进入公共长缓存，Next 服务端请求保持 cache: "no-store"。权限变更后再次加载页面会重新走登录态和权限判断， 不会因为 CDN 或 Redis 列表缓存导致越权入口长期存在。

场景 D：Redis 故障

缓存是性能优化，不是业务事实来源。Redis 读写失败时， CommodityCacheService 会记录结构化错误日志，并让列表请求降级为直接查询 backend；页面慢一些，但功能可用。
