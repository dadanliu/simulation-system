# 17. 结构化日志与链路排查

本文记录 `8.1 结构化日志` 的落地方案。

目标不是“多打一行日志”，而是让日志字段稳定、可搜索、能按 `traceId` 串起一次请求在 BFF 和 backend 的完整链路。

## 17.1 图例

```text
Client
  |
  | x-trace-id: trace-create-001
  v
BFF
  |
  +-- http_request_completed
  +-- backend_request_started
  +-- backend_request_completed
  |
  | x-trace-id: trace-create-001
  v
Backend
  |
  +-- http_request_completed
```

一次商品创建可以按同一个 `traceId` 搜索到：

```text
BFF 接收请求
BFF 调用 backend
Backend 处理请求
BFF 返回响应
```

## 17.2 JSON Log 字段

请求成功日志：

```json
{
  "timestamp": "2026-05-09T03:39:40.800Z",
  "level": "info",
  "service": "bff",
  "context": "RequestLoggingInterceptor",
  "event": "http_request_completed",
  "method": "POST",
  "path": "/api/commodity/create",
  "status": 201,
  "durationMs": 42,
  "traceId": "trace-create-001"
}
```

BFF 调 backend 日志：

```json
{
  "timestamp": "2026-05-09T03:39:40.760Z",
  "level": "info",
  "service": "bff",
  "context": "ApiClientService",
  "event": "backend_request_completed",
  "method": "POST",
  "path": "/api/commodity/create",
  "status": 201,
  "durationMs": 18,
  "traceId": "trace-create-001"
}
```

Backend 请求日志：

```json
{
  "timestamp": "2026-05-09T03:39:40.770Z",
  "level": "info",
  "service": "backend",
  "context": "RequestLoggingInterceptor",
  "event": "http_request_completed",
  "method": "POST",
  "path": "/api/commodity/create",
  "status": 201,
  "durationMs": 12,
  "traceId": "trace-create-001"
}
```

字段说明：

| 字段         | 含义                         |
| ------------ | ---------------------------- |
| `timestamp`  | 日志时间                     |
| `level`      | `info` / `warn` / `error`    |
| `service`    | `bff` 或 `backend`           |
| `context`    | 产生日志的类                 |
| `event`      | 稳定事件名，便于日志平台过滤 |
| `method`     | HTTP method                  |
| `path`       | 请求路径                     |
| `status`     | HTTP 状态码                  |
| `durationMs` | 请求耗时                     |
| `traceId`    | 链路追踪 ID                  |

## 17.3 错误日志

500 错误日志会带异常类型、错误信息和 stack：

```json
{
  "timestamp": "2026-05-09T03:39:40.800Z",
  "level": "error",
  "service": "bff",
  "context": "HttpExceptionFilter",
  "event": "http_request_failed",
  "method": "POST",
  "path": "/api/commodity/create",
  "status": 500,
  "durationMs": 12,
  "traceId": "trace-create-001",
  "errorName": "TypeError",
  "errorMessage": "Cannot read properties of undefined",
  "stack": "TypeError: Cannot read properties of undefined\n    at ..."
}
```

4xx 错误使用 `warn`：

```json
{
  "level": "warn",
  "service": "bff",
  "context": "HttpExceptionFilter",
  "event": "http_request_failed",
  "method": "GET",
  "path": "/api/commodity/missing",
  "status": 404,
  "durationMs": 8,
  "traceId": "trace-detail-missing",
  "errorName": "NotFoundException",
  "errorMessage": "commodity not found"
}
```

## 17.4 慢接口定位

因为每条请求日志都有 `durationMs`，日志平台可以直接筛选慢接口：

```text
event = http_request_completed
durationMs > 1000
```

图例：

```text
慢接口告警
  |
  v
按 durationMs 排序
  |
  v
拿到 traceId
  |
  v
查同 traceId 的 BFF / backend 日志
  |
  v
定位慢在 BFF、backend、数据库还是外部依赖
```

## 17.6 验收对应

```text
[x] 一次商品创建可用 traceId 串起 BFF 和 backend 日志
    BFF ApiClient 会把 x-trace-id 转发给 backend，双方日志都带 traceId。

[x] 500 错误能查到 stack
    HttpExceptionFilter 对非 HttpException 的 Error 输出 errorName / errorMessage / stack。

[x] 慢接口能看到 durationMs
    RequestLoggingInterceptor 和 ApiClientService 都输出 durationMs。

[x] 日志字段稳定可搜索
    日志输出为 JSON line，固定 level / service / context / event / traceId 等字段。
```
