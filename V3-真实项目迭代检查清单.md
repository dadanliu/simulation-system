# Stage 3: 真实项目生产化迭代 Checklist

目标：把当前 `next-bff` 从学习型后台系统，迭代成更接近真实项目的系统。  
定位：资深前端主导全栈协作，不要求前端独自完成所有后端/运维能力，但要能拆任务、定边界、推动落地、定义验收。

当前系统：

```text
apps/client  -> Next.js 管理后台
apps/bff     -> NestJS BFF，负责登录、会话、权限、响应结构、接口聚合
apps/server  -> NestJS backend，负责商品、上传、持久化
MongoDB      -> users / roles / permissions / commodities / audit_logs
```

读法：

- `功能`：真实项目里要补齐的系统能力
- `能力`：完成这个功能涉及的核心知识点
- `验收`：做完后如何判断达到可交付标准

---

## 1. 登录、会话与账号安全生产化

登录是后台系统的第一道边界，优先级最高。当前系统已经有 cookie session 登录链路，但还没达到真实项目安全标准。

### 1.1 密码从明文升级为 hash 存储

- [ ] 功能：用户密码不再明文保存
      能力：bcrypt / argon2、salt、passwordHash 字段、敏感字段脱敏
- [ ] 功能：登录时使用 hash compare 校验密码
      能力：认证流程、异步密码校验、统一失败提示
- [ ] 功能：seed 初始化测试账号时写入 hash
      能力：可重复 seed、幂等初始化、测试账号管理
- [ ] 功能：用户查询接口永远不返回 passwordHash
      能力：DTO / ViewModel / 数据脱敏

验收：

- [ ] MongoDB 中不存在明文密码
- [ ] `admin / admin123` 仍可登录
- [ ] 错误账号和错误密码都返回统一 401
- [ ] 任意用户接口响应不包含 password / passwordHash

### 1.2 session 从内存升级为可持久化存储

- [ ] 功能：BFF 重启后登录态不丢
      能力：Redis session、session TTL、服务重启恢复
- [ ] 功能：退出登录后 session 立即失效
      能力：session 删除、cookie 清理、服务端会话状态
- [ ] 功能：session 过期后访问受保护接口返回 401
      能力：过期策略、401 处理、登录回跳
- [ ] 功能：支持单用户多端登录
      能力：sessionId、userId、设备维度、会话列表

验收：

- [ ] 登录后重启 BFF，刷新页面仍能识别当前用户
- [ ] 退出后访问商品列表 API 返回 401
- [ ] Redis 中能看到 session TTL
- [ ] session 过期后前端跳转 `/login?next=...`

### 1.3 cookie 安全策略完善

- [ ] 功能：生产环境 cookie 自动带 `Secure`
      能力：环境变量、ConfigModule、HTTPS cookie
- [ ] 功能：明确 `HttpOnly / SameSite / Max-Age / Path`
      能力：浏览器 cookie 安全模型
- [ ] 功能：根据环境配置 `COOKIE_SECURE`
      能力：本地、测试、生产配置差异
- [ ] 功能：登录和退出使用一致 cookie 策略
      能力：Set-Cookie、清 cookie、属性一致性

验收：

- [ ] 本地 HTTP 环境 cookie 可正常登录
- [ ] 生产配置下 Set-Cookie 包含 `Secure`
- [ ] JS 无法读取 `next_bff_session`
- [ ] 退出登录后 cookie 被清理

### 1.4 CSRF 防护

- [ ] 功能：所有高风险写操作具备 CSRF 防护
      能力：cookie session 风险、SameSite、CSRF token、double-submit
- [ ] 功能：前端写请求携带 CSRF token
      能力：请求封装、header 注入、token 生命周期
- [ ] 功能：BFF 校验 CSRF token
      能力：Guard / middleware、403 响应、错误处理
- [ ] 功能：登录、退出、创建、删除、状态变更纳入风险评估
      能力：写操作分类、安全边界

验收：

- [ ] 无 CSRF token 的写请求被拒绝
- [ ] 正常页面操作不受影响
- [ ] 403 返回统一错误结构
- [ ] 能解释 SameSite 和 CSRF token 各自解决什么问题

### 1.5 登录风控基础

- [ ] 功能：登录失败次数限制
      能力：rate limit、账号维度、IP 维度
- [ ] 功能：多次失败后短时间锁定
      能力：锁定策略、错误提示、安全与体验平衡
- [ ] 功能：登录日志记录
      能力：安全审计、operator、IP、User-Agent、traceId
- [ ] 功能：前端展示统一失败提示
      能力：避免账号枚举、敏感信息保护

验收：

- [ ] 连续错误登录触发限流或锁定
- [ ] 前端不提示“账号不存在”
- [ ] 登录日志可按用户和时间查询
- [ ] 限流响应结构统一且前端能稳定展示

## 9. 性能、缓存与数据规模

复杂缓存和数据规模是真实项目必须考虑的方向。资深前端不需要运维 Redis 集群或 MongoDB 分片，但必须知道它们如何影响页面体验和接口设计。

### 9.1 多层缓存设计

- [x] 功能：静态资源走浏览器缓存和 CDN
      能力：Cache-Control、ETag、CDN、资源 hash
- [x] 功能：图片走 CDN 长缓存
      能力：图片版本化、缩略图、多尺寸
- [x] 功能：商品列表可短缓存
      能力：TTL、stale-while-revalidate、缓存失效
- [x] 功能：权限和审计数据谨慎缓存
      能力：安全数据、强一致需求
- [x] 功能：写操作后失效相关缓存
      能力：cache key、invalidate、router refresh

验收：

- [x] 能画出 Browser / CDN / Next / BFF / Redis / DB 缓存层级
- [x] 创建商品后列表不会长期展示旧数据
- [x] 删除商品后列表不可见
- [x] 权限变更不会因为缓存导致越权入口长期可见

### 9.2 缓存问题排障

- [x] 功能：判断旧数据来自浏览器、CDN、Next、BFF、Redis 还是 DB
      能力：Network、response header、traceId、日志

      能力：热点数据、回源、降级
- [x] 功能：缓存异常时前端可降级
      能力：loading、重试、刷新、兜底文案
- [x] 功能：大促或发布前可配合缓存预热
      能力：预热、回源压力、发布稳定性

验收：

- [x] 能解释“浏览器没发请求”和“服务端返回旧数据”的区别
