# Stage 2: 资深前端生产级能力深度 Checklist

目标：基于当前 `next-bff` 系统，挑选资深前端需要真正掌握的生产级能力。  
重点不是把 backend 做到极致，而是让前端能讲清楚：安全边界、请求链路、错误治理、可观测性、权限体验、上传体验、性能与工程质量。

当前系统：

```text
apps/client  -> Next.js 管理后台页面
apps/bff     -> NestJS BFF，负责登录、会话、权限、响应结构、接口聚合
apps/server  -> NestJS backend，负责商品、上传、持久化
MongoDB      -> users / roles / permissions / commodities / audit_logs
```

范围说明：

- [ ] 复杂缓存体系需要考虑，但资深前端重点掌握“缓存层级、失效策略、一致性、用户体验和排障”，不要求实现 Redis 集群
- [ ] MongoDB 分片需要了解，但资深前端重点掌握“为什么会影响查询、分页、排序、接口契约和故障表现”，不要求达到 DBA 深度
- [ ] 前端要理解 MongoDB 索引、分页、schema 演进、分片键选择对接口体验的影响
- [ ] 前端要能和后端讨论安全、稳定性、可观测性，而不是只写页面

读法：

- `功能` 是当前系统应该补齐或能解释的生产级行为
- `前端能力` 是资深前端需要掌握的知识点
- `BFF / backend 协作` 是前端需要和服务端对齐的接口、协议、边界
- `验收` 是怎么确认你真的掌握，而不是只会背概念

## 1. 登录、会话与浏览器安全

### 1.1 登录态为什么用 httpOnly cookie

- [ ] 功能：登录成功后浏览器持有 `next_bff_session`
  前端能力：理解 cookie、httpOnly、SameSite、Secure、Max-Age 的区别
- [ ] 功能：Client Component 同源请求自动携带 cookie
  前端能力：理解浏览器 credential 规则和同源请求行为
- [ ] 功能：Server Component 请求 BFF 时手动透传 cookie
  前端能力：理解 Next.js server runtime 不会自动继承浏览器 cookie
- [ ] 功能：未登录访问页面跳转登录页
  前端能力：理解 server-side redirect 与 client-side router push 的差异
- [ ] 功能：登录失败不暴露敏感信息
  BFF / backend 协作：统一 401，message 不泄露账号是否存在

验收：

- [ ] 能解释为什么不用 localStorage 存 token
- [ ] 能解释为什么 `next/headers` 只在服务端使用
- [ ] 能解释为什么 Server Component 里要写 `headers: { cookie }`
- [ ] 能说清楚 httpOnly 防什么、不防什么

### 1.2 CSRF 与 SameSite 策略

- [ ] 功能：后台管理站点使用 cookie session
  前端能力：理解 cookie 自动携带带来的 CSRF 风险
- [ ] 功能：生产环境 cookie 使用 `Secure`
  前端能力：理解 HTTPS 与 Secure cookie 的关系
- [ ] 功能：根据跨站需求选择 SameSite
  前端能力：区分 `Lax`、`Strict`、`None`
- [ ] 功能：高风险写操作考虑 CSRF token 或 double-submit
  BFF / backend 协作：BFF 校验 CSRF token，前端在写请求携带 token
- [ ] 功能：登录、删除、状态变更等接口都属于写操作
  前端能力：识别哪些交互必须纳入 CSRF 风险评估

验收：

- [ ] 能解释为什么 cookie session 比 bearer token 更容易遇到 CSRF
- [ ] 能解释 SameSite=Lax 能挡住哪些场景，挡不住哪些场景
- [ ] 能设计一个后台管理系统的 CSRF 防护方案

### 1.3 密码与账号安全的前端边界

- [ ] 功能：登录表单不在前端做“账号是否存在”的提示
  前端能力：避免用户枚举风险
- [ ] 功能：密码输入不写入日志、不进入 URL query
  前端能力：理解浏览器历史、日志、埋点中的敏感信息泄露
- [ ] 功能：登录失败给稳定通用提示
  前端能力：安全提示与用户体验平衡
- [ ] 功能：密码 hash 由 backend 完成
  BFF / backend 协作：前端知道 bcrypt / argon2 是服务端存储策略，不在浏览器实现

验收：

- [ ] 能解释为什么前端不负责密码 hash
- [ ] 能指出登录表单里哪些字段不能进埋点
- [ ] 能设计登录失败、锁定、限流的前端提示策略

## 2. 权限模型与前端体验

### 2.1 前端隐藏按钮不是权限

- [ ] 功能：operator 不显示审计日志入口
  前端能力：根据当前用户角色做体验层裁剪
- [ ] 功能：operator 手动访问审计页时展示无权限页面
  前端能力：处理 403，不让 Next runtime overlay 暴露给用户
- [ ] 功能：BFF Guard 兜底权限
  BFF / backend 协作：前端不能依赖隐藏按钮作为安全边界
- [ ] 功能：不同角色看到不同操作入口
  前端能力：角色驱动导航、按钮、空状态

验收：

- [ ] 能解释“隐藏按钮是 UX，不是安全”
- [ ] 能解释为什么 403 要做友好页而不是跳登录
- [ ] 能说清楚 admin / operator / viewer 在页面上的差异

### 2.2 Guard + decorator 对前端意味着什么

- [ ] 功能：BFF 接口声明权限点，例如 `commodity:update`
  前端能力：理解接口能力和页面能力的映射
- [ ] 功能：前端根据用户角色只展示可用入口
  前端能力：权限驱动 UI，不把权限字符串散落到组件各处
- [ ] 功能：接口 403 统一展示
  前端能力：统一错误处理策略
- [ ] 功能：权限变更后页面能刷新当前用户能力
  BFF / backend 协作：`/api/auth/me` 返回角色或权限摘要

验收：

- [ ] 能解释 decorator 是声明权限，Guard 是执行权限
- [ ] 能画出“页面按钮 -> BFF 权限点 -> Guard 校验”的关系
- [ ] 能判断一个权限错误应该在页面层、组件层还是请求层处理

## 3. 统一响应、统一异常与前端数据层

### 3.1 前端稳定读取统一成功响应

- [ ] 功能：所有成功响应统一为 `success / data / message / traceId`
  前端能力：封装通用 `readApiResponse`
- [ ] 功能：Controller 不手写 `{ success: true }`
  前端能力：知道响应结构由 BFF Interceptor 保证
- [ ] 功能：前端只依赖稳定协议，不依赖 backend 原始结构
  BFF / backend 协作：BFF 做 backend 协议适配
- [ ] 功能：文件流等特殊场景允许跳过包装
  前端能力：理解 JSON API 与 binary response 的差异

验收：

- [ ] 能解释为什么统一成功响应放 Interceptor
- [ ] 能写出一个泛型 `ApiResponse<T>` 类型
- [ ] 能解释为什么前端不应该每个页面重复判断响应结构

### 3.2 前端稳定处理统一错误响应

- [ ] 功能：400 / 401 / 403 / 404 / 500 响应结构一致
  前端能力：区分参数错误、未登录、无权限、不存在、系统错误
- [ ] 功能：401 跳登录并保留 next path
  前端能力：登录后回跳
- [ ] 功能：403 显示无权限页面或禁用操作
  前端能力：权限错误不等于登录失效
- [ ] 功能：404 显示资源不存在
  前端能力：动态路由错误边界
- [ ] 功能：500 显示系统错误和 traceId
  前端能力：用户可反馈 traceId 给排障

验收：

- [ ] 能解释为什么统一异常放 Filter
- [ ] 能写出错误处理分支：401 / 403 / 404 / default
- [ ] 能解释为什么前端错误页要展示 traceId

### 3.3 Server Component 数据读取边界

- [ ] 功能：列表页 SSR 读取商品列表
  前端能力：理解 Server Component 适合首屏数据
- [ ] 功能：创建、删除、状态变更由 Client Component 发起
  前端能力：理解用户交互需要 client runtime
- [ ] 功能：Server Component 请求需要手动透传 cookie
  前端能力：理解请求上下文
- [ ] 功能：Client Component 请求同源 API 自动带 cookie
  前端能力：理解浏览器请求模型

验收：

- [ ] 能判断一个数据请求应该放 Server Component 还是 Client Component
- [ ] 能解释为什么 `server.ts` 文件在 client app 里但运行在 Next server
- [ ] 能解释 Next rewrite 在当前系统中的作用

## 5. 商品列表、分页与性能

### 5.1 offset 分页的前端风险

- [ ] 功能：商品列表支持 page / pageSize
  前端能力：维护 URL query，支持刷新和分享
- [ ] 功能：pageSize 有最大值
  BFF / backend 协作：DTO 限制，前端不信任自己
- [ ] 功能：offset 分页在大数据量下可能变慢
  前端能力：知道页码越深越慢的原因
- [ ] 功能：数据频繁插入时可能出现翻页漂移
  前端能力：理解重复、漏读、顺序抖动

验收：

- [ ] 能解释 offset 分页为什么大 offset 慢
- [ ] 能解释什么时候要升级 cursor 分页
- [ ] 能设计列表 URL query 与分页组件状态同步

### 5.2 cursor 分页什么时候需要

- [ ] 功能：按创建时间稳定向后翻页
  前端能力：理解 cursor 是“从某个锚点继续查”
- [ ] 功能：列表数据持续增长时避免翻页漂移
  BFF / backend 协作：cursor 可由 `{ createdAt, id }` 组成
- [ ] 功能：前端不再展示任意页跳转
  前端能力：理解 cursor 分页的交互取舍
- [ ] 功能：支持“加载更多”或“下一页”
  前端能力：设计 cursor UX

验收：

- [ ] 能比较 page/offset 和 cursor 的优缺点
- [ ] 能解释为什么 cursor 分页不适合“跳到第 100 页”
- [ ] 能画出 cursor 请求和响应结构

### 5.3 图片展示性能

- [ ] 功能：列表展示商品缩略图
  前端能力：缩略图尺寸、占位、fallback
- [ ] 功能：详情展示商品大图
  前端能力：图片尺寸、object-fit、布局稳定性
- [ ] 功能：生产图片走 CDN

## 6. 上传能力

### 6.1 商品图片上传链路

- [ ] 功能：用户选择 JPG / PNG / WEBP
  前端能力：accept 只是体验提示，不是安全校验
- [ ] 功能：前端限制 2MB
  前端能力：提前反馈，减少无效请求
- [ ] 功能：BFF 再次校验文件类型和大小
  BFF / backend 协作：后端兜底
- [ ] 功能：上传成功返回文件元数据
  前端能力：保存 `fileId / url / mimeType / size`
- [ ] 功能：创建商品时提交图片元数据
  前端能力：上传和创建是两个动作，需要状态串联

验收：

- [ ] 能解释为什么前端 accept 不可信
- [ ] 能解释为什么商品表只保存文件元数据，不保存二进制
- [ ] 能解释上传成功但创建失败时如何处理孤儿文件

### 6.2 生产上传还差什么

- [ ] 功能：真实对象存储，例如 S3 / OSS
  前端能力：理解直传、预签名 URL、服务端中转的区别
- [ ] 功能：文件内容嗅探
  BFF / backend 协作：不能只相信 MIME type 和后缀
- [ ] 功能：病毒扫描或安全扫描
  前端能力：知道上传是安全风险入口
- [ ] 功能：私有文件访问控制
  前端能力：理解公开 URL、签名 URL、鉴权代理
- [ ] 功能：上传进度、失败重试、取消上传
  前端能力：复杂交互体验

验收：

- [ ] 能比较“浏览器直传对象存储”和“先传 BFF 再传 backend”
- [ ] 能说明大文件上传为什么需要分片或断点续传
- [ ] 能列出上传安全检查清单

## 10. 复杂缓存体系

### 10.1 资深前端需要理解哪些缓存层

- [ ] 功能：浏览器 HTTP 缓存控制静态资源
  前端能力：理解 `Cache-Control`、ETag、强缓存、协商缓存
- [ ] 功能：CDN 缓存图片、JS、CSS 等静态资源
  前端能力：理解 CDN 命中、回源、刷新、预热
- [ ] 功能：Next.js 可能存在 fetch cache、route cache、full route cache
  前端能力：理解 `cache: "no-store"`、动态渲染、revalidate 的影响
- [ ] 功能：BFF 可以缓存低频变化的数据
  前端能力：理解 BFF cache 和 client state cache 的边界
- [ ] 功能：backend / Redis 可以缓存热点查询
  前端能力：理解接口返回快不代表数据一定最新

验收：

- [ ] 能画出 Browser Cache / CDN / Next Cache / BFF Cache / Redis / DB 的层级
- [ ] 能解释为什么缓存越多，性能可能越好，但一致性和排障更复杂
- [ ] 能说明当前商品后台哪些数据适合缓存，哪些不适合缓存

### 10.2 当前系统哪些地方适合缓存，哪些不适合

- [ ] 功能：登录态、当前用户权限不能长期缓存
  前端能力：权限变更后页面需要及时反映
- [ ] 功能：商品列表可以短时间缓存，但写操作后必须失效
  前端能力：创建、删除、状态变更后刷新列表或失效列表缓存
- [ ] 功能：商品详情可以按 id 短缓存
  前端能力：状态变更、删除后刷新详情或跳转
- [ ] 功能：审计日志默认不做强缓存
  前端能力：审计页面强调可信和及时，不追求极致缓存命中
- [ ] 功能：图片资源适合 CDN 长缓存
  前端能力：图片 URL 指纹化或版本化，避免换图后仍显示旧图

验收：

- [ ] 能把 `auth/me`、商品列表、商品详情、审计日志、图片分别归类为是否适合缓存
- [ ] 能解释为什么商品写操作后要 invalidate 列表和详情
- [ ] 能说明权限数据缓存过久会带来什么安全问题

### 10.3 缓存一致性与前端体验

- [ ] 功能：创建商品后列表要能看到新商品
  前端能力：写后读一致性、router refresh、乐观更新或重新拉取
- [ ] 功能：删除商品后列表不可见
  前端能力：删除成功后清理本地列表状态或重新请求
- [ ] 功能：状态变更后详情页和列表状态一致
  前端能力：避免页面 A 显示 on_sale，页面 B 仍显示 pending
- [ ] 功能：上传图片后列表展示新图
  前端能力：图片缓存失效、URL 版本化、缩略图刷新
- [ ] 功能：缓存命中但数据过期时给用户可理解反馈
  前端能力：stale-while-revalidate、刷新按钮、更新时间提示

验收：

- [ ] 能解释 strong consistency、eventual consistency、stale data 对用户体验的区别
- [ ] 能设计“删除成功但列表仍显示旧数据”的排查路径
- [ ] 能说明什么时候适合乐观更新，什么时候必须以后端结果为准

### 10.4 缓存失效策略

- [ ] 功能：按 key 失效商品列表
  前端能力：知道列表 key 通常包含 filter / page / sort
- [ ] 功能：按 id 失效商品详情
  前端能力：详情 key 与商品 id 绑定
- [ ] 功能：写操作后批量失效相关 query
  前端能力：创建影响列表，删除影响列表和详情，状态变更影响列表和详情
- [ ] 功能：权限变化后失效当前用户和菜单缓存
  前端能力：避免用户角色变更后仍看到旧入口
- [ ] 功能：缓存失效失败时有兜底
  前端能力：手动刷新、短 TTL、错误提示

验收：

- [ ] 能设计商品列表 cache key：`commodity:list:{filters}:{page}:{sort}`
- [ ] 能列出创建、删除、状态变更分别应该失效哪些缓存
- [ ] 能解释 TTL 不是完整的一致性方案，只是兜底

### 10.5 复杂缓存体系的排障能力

- [ ] 功能：页面数据旧，要判断旧在浏览器、CDN、Next、BFF、Redis 还是 DB
  前端能力：分层排查缓存命中来源
- [ ] 功能：Network 面板查看 `cache-control`、status、response headers
  前端能力：识别 from disk cache、from memory cache、304
- [ ] 功能：traceId 关联 BFF/backend 日志
  前端能力：判断请求有没有真正打到服务端
- [ ] 功能：缓存导致权限或审计异常时能快速降级
  前端能力：关闭缓存、缩短 TTL、强制刷新

验收：

- [ ] 能说出“浏览器没发请求”和“服务端返回旧数据”的区别
- [ ] 能用 Network + traceId 判断请求是否命中服务端
- [ ] 能设计一个“商品状态变了但列表没变”的排查 checklist

### 10.6 资深前端不需要深挖但要能沟通的缓存后端概念

- [ ] 功能：Redis 作为热点数据缓存
  前端能力：理解 Redis 是服务端缓存，不是浏览器缓存
- [ ] 功能：缓存穿透、击穿、雪崩
  前端能力：知道这些问题会表现为接口突然变慢或大量 500
- [ ] 功能：分布式锁或 singleflight 防止热点回源
  前端能力：知道它解决“同一时间大量请求打 DB”
- [ ] 功能：缓存预热和 CDN 预热
  前端能力：知道大促或发布前为什么要预热
- [ ] 功能：缓存降级
  前端能力：知道缓存系统异常时页面应该能降级展示

验收：

- [ ] 能用非后端术语解释穿透、击穿、雪崩
- [ ] 能说明这些缓存事故在前端页面上的表现
- [ ] 能和后端讨论是否需要缓存，而不是默认所有接口都缓存

## 11. MongoDB 分片、索引与数据规模

### 11.1 资深前端为什么需要了解 MongoDB 分片

- [ ] 功能：商品数据规模增大后单库单集合可能成为瓶颈
  前端能力：理解数据规模会影响接口延迟和分页体验
- [ ] 功能：分片会改变查询性能特征
  前端能力：知道查询是否带 shard key 会影响是否广播查询
- [ ] 功能：跨分片排序和分页成本更高
  前端能力：理解深分页、全局排序在大规模数据下更难
- [ ] 功能：分片迁移期间可能出现延迟抖动
  前端能力：页面要有 loading、重试、错误兜底

验收：

- [ ] 能解释为什么资深前端需要懂一点分片，但不需要会运维分片集群
- [ ] 能说清楚分片对列表页查询、排序、分页的影响
- [ ] 能从用户反馈“列表偶发很慢”联想到数据层热点或跨分片查询

### 11.2 shard key 对接口设计的影响

- [ ] 功能：商品按租户、店铺、业务线隔离时可能选择 `tenantId` / `shopId` 作为 shard key
  前端能力：知道请求里需要稳定携带租户上下文
- [ ] 功能：只按 `createdAt` 查询可能导致跨分片扫描
  前端能力：理解筛选条件缺少 shard key 的风险
- [ ] 功能：列表接口应尽量包含高选择性条件
  前端能力：页面筛选项和后端查询能力要对齐
- [ ] 功能：全局搜索可能需要搜索系统而不是直接扫 MongoDB
  前端能力：理解 keyword search 的规模边界

验收：

- [ ] 能解释 shard key 是什么，以及为什么选错会导致热点
- [ ] 能说明前端 URL query / 请求参数为什么可能需要带 `tenantId`
- [ ] 能判断一个列表查询是否可能变成跨分片查询

### 11.3 分片下的分页与排序

- [ ] 功能：offset 分页在大数据量和分片下成本更高
  前端能力：理解深页查询更慢，跨分片更明显
- [ ] 功能：cursor 分页更适合大规模时间线列表
  前端能力：理解 cursor 需要稳定排序键
- [ ] 功能：排序字段必须和索引、分片策略协同
  前端能力：不是所有 UI 排序都应该开放
- [ ] 功能：分页交互可能从“跳页”改成“下一页 / 加载更多”
  前端能力：能解释交互变化背后的性能原因

验收：

- [ ] 能解释为什么分片后“按任意字段排序 + 跳到第 N 页”会很贵
- [ ] 能设计 cursor 分页的前端状态结构
- [ ] 能说服产品为什么某些排序字段不开放

### 11.4 索引与查询计划前端要懂到什么程度

- [ ] 功能：状态筛选需要 status 索引
  前端能力：知道筛选项不是随便加，背后需要索引支持
- [ ] 功能：创建时间排序需要 createdAt 索引
  前端能力：知道默认排序要稳定
- [ ] 功能：状态 + 创建时间组合查询需要复合索引
  前端能力：知道组合筛选和排序需要一起考虑
- [ ] 功能：低选择性字段不一定适合单独索引
  前端能力：理解“有索引”不等于“一定快”
- [ ] 功能：查询计划异常会导致页面慢
  前端能力：能向后端提供 query、page、sort、traceId 便于分析

验收：

- [ ] 能解释每个商品索引为什么存在
- [ ] 能说出新增筛选项时要问后端哪些问题
- [ ] 能区分“前端渲染慢”和“DB 查询慢”的排查信号

### 11.5 分片和多租户对前端权限的影响

- [ ] 功能：用户只能看到自己租户或组织的数据
  前端能力：租户上下文不能只存在前端状态里
- [ ] 功能：BFF / backend 必须从 session 或 token 中兜底租户权限
  前端能力：不能让用户通过改 query 越权访问其他租户
- [ ] 功能：审计日志按租户隔离
  前端能力：审计页面也要带租户上下文
- [ ] 功能：分片键和权限边界最好一致
  前端能力：理解 tenantId 既影响权限，也影响查询路由

验收：

- [ ] 能解释为什么不能只靠前端传 `tenantId`
- [ ] 能说明多租户系统里列表、详情、审计都要做租户隔离
- [ ] 能讲清楚 shard key 和权限边界一致的好处

### 11.6 MongoDB 分片故障在前端的表现

- [ ] 功能：某些租户访问慢，其他租户正常
  前端能力：可能是热点分片或数据倾斜
- [ ] 功能：某些查询条件慢，其他条件正常
  前端能力：可能是跨分片查询或缺索引
- [ ] 功能：分页越往后越慢
  前端能力：offset 深分页问题
- [ ] 功能：写操作偶发慢
  前端能力：可能是 chunk migration、锁、写关注配置
- [ ] 功能：接口错误率升高
  前端能力：结合 traceId、status、durationMs 给后端定位

验收：

- [ ] 能把“页面慢”拆成前端资源慢、接口慢、DB 慢、分片热点几类
- [ ] 能提供排障信息：用户、租户、URL query、traceId、时间点、复现步骤
- [ ] 能说明为什么前端需要 loading、重试、降级，而不是只等后端优化
